## Summary

Build a client-side-only flashcard web app with spaced repetition scheduling. Users create Decks containing Cards, practice in daily sessions, and rate recall quality (Again / Hard / Good / Easy) so the SM-2 algorithm can schedule future reviews. All data is persisted to browser localStorage; no server or database is required. The UI is built with TailwindCSS using a mobile-first, minimalist design.

### Phase 0 ‚Äî Foundation & Contracts *(enables test writing)*

Goal: all interfaces, types, and the SM-2 core are defined so tests can be written first.

| # | Deliverable | Contract |
|---|---|---|
| 0.1 | `src/lib/types.ts` | `Deck`, `Card`, `ReviewState`, `Session` TypeScript interfaces |
| 0.2 | `src/lib/sm2.ts` | Pure function `schedule(state, rating) ‚Üí ReviewState`; deterministic, no side-effects |
| 0.3 | `src/lib/storage.ts` | `StorageAdapter` interface: `getDecks`, `saveDecks`, `getCards`, `saveCards`, `getReviewStates`, `saveReviewStates` |

**Contract ‚Äî SM-2 `schedule` function interface:**

```typescript
type Rating = 'again' | 'hard' | 'good' | 'easy';

interface ReviewState {
  cardId: string;
  interval: number;       // days until next review
  easeFactor: number;     // starts at 2.5
  repetitions: number;    // consecutive successful reviews
  dueDate: string;        // ISO 8601 date string
  lapses: number;         // times rated 'again' after being learned
}

function schedule(state: ReviewState, rating: Rating): ReviewState;
```

**Contract ‚Äî Storage namespace keys:**

```
localStorage key pattern:  flashcard_v1_decks
                           flashcard_v1_cards_{deckId}
                           flashcard_v1_review_states
```

## Data Model

### `Deck`

```typescript
interface Deck {
  id: string;          // uuid v4
  name: string;
  createdAt: string;   // ISO 8601
}
```

### `Card`

```typescript
interface Card {
  id: string;          // uuid v4
  deckId: string;
  front: string;
  back: string;
  createdAt: string;
  updatedAt: string;
}
```

### `ReviewState` (one per Card)

```typescript
interface ReviewState {
  cardId: string;
  interval: number;        // days; 0 = new/unlearned
  easeFactor: number;      // default 2.5
  repetitions: number;     // consecutive non-"again" ratings
  dueDate: string;         // ISO 8601 date; new cards ‚Üí today
  lapses: number;
}
```

### SM-2 Scheduling Rules

| Rating | Effect |
|---|---|
| **Again** | `interval = 1`, `repetitions = 0`, `lapses++`, `easeFactor = max(1.3, ef - 0.2)` |
| **Hard** | `interval = interval √ó 1.2`, `easeFactor = max(1.3, ef - 0.15)` |
| **Good** | Standard SM-2: `interval = interval √ó easeFactor` (min 1 day step) |
| **Easy** | `interval = interval √ó easeFactor √ó 1.3`, `easeFactor += 0.15` |
| New card (first review) | `interval = 1` (Hard/Again) or `interval = 4` (Good) or `interval = 7` (Easy) |

## Key Architectural Decisions

| Decision | Choice | Rationale |
|---|---|---|
| State management | React hooks + localStorage (no Redux/Zustand) | Simplicity; single-user app with no shared state across tabs |
| Routing | React Router v6 (hash-based) | Works without a server; no build-time config needed |
| SM-2 implementation | Custom pure function | Minimal dependency; algorithm is small and well-specified |
| Styling | TailwindCSS utility classes only | No CSS files to maintain; design system enforced by config |
| Build tool | Vite | Fast HMR; zero-config TypeScript; modern ES output |
